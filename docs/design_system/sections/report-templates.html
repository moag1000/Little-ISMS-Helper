Output · ISMS-Report-Dokumente

Report-Templates

Jeder ISMS-Report braucht dieselben Bausteine — Cover · Executive-Summary · Daten-Sections · Appendix · Footer-Disclaimer — sonst ist er weder vorstandstauglich noch audit-fest. Statt jeden Report ad-hoc neu zu layouten (~30 Templates derzeit), kommt jeder aus derselben Vorlage: <article class="report-doc"> mit semantischen Sub-Komponenten.

Der Werkzeugkasten unten ist ein Markup-System, kein One-Page-Render. Jeder Report ist ein eigenes Dokument — aber alle nutzen dieselben Klassen, Header, KPI-Tiles, Watermarks und Disclaimer-Footer. So ist „Board-One-Pager" optisch nicht von „NIS2-Bericht" zu unterscheiden — beabsichtigt: einheitliches Erscheinungsbild für die gesamte Report-Familie.

Konsistent
Eine Vorlage, alle Reports. Cover & Footer immer gleich.
Vorstands-tauglich
Executive-Summary mit TL;DR + KPI-Tiles ganz vorne.
Audit-fest
Datenquellen, Methodik & Glossar im Appendix Pflicht.
Druckbar
A4 portrait + landscape-toggle, repeating Header, Status-Watermark.
Schwester-Vorlage zum Policy-Doc. Während policy-doc ISMS-Richtlinien rendert (Cover · TOC · History · Body · Signature), rendert report-doc Berichte (Cover · Exec · Data · Appendix · Footer). Beide nutzen dieselben Aurora-Tokens, dieselbe Status-Watermark-Logik und dieselbe Print-Pipeline.
Report · Anatomie

Aufbau eines Report-Dokuments

Ein Report ist ein <article class="report-doc"> mit fünf Pflicht-Slots: .report-doc__cover, .report-doc__exec-summary, .report-doc__data-section (1..n, wiederholbar), .report-doc__appendix, .report-doc__footer-disclaimer. Reihenfolge ist verbindlich — die Print-Styles setzen Page-Breaks an genau diesen Übergängen.

1
.report-doc__cover
Logo · Titel · Subtitle (z.B. „Q2/2026") · Status · Generated-At · Audience-Pill
›
2
.report-doc__exec-summary
TL;DR · 3-5 Key-Findings · 4-up KPI-Tiles · Trend-Indicator
›
3
.report-doc__data-section ×n
Heading · Intro · Charts (Bar/Line/Pie/Heatmap) · Tabelle
›
4
.report-doc__appendix
Methodik · Datenquellen · Glossar
›
5
.report-doc__footer-disclaimer
Verteilung · Vertraulichkeits-Klausel · Branding · Page-Number
<article class="report-doc report-doc--final report-doc--vorstand"
         data-report-status="final"
         data-report-audience="vorstand">

  <header  class="report-doc__cover">…</header>
  <section class="report-doc__exec-summary">…</section>

  <section class="report-doc__data-section">
    <h2 class="report-doc__section-heading">Risiko-Verteilung</h2>
    <div class="report-doc__chart-grid">…charts…</div>
    <div class="report-doc__table-wrap">…tabelle…</div>
  </section>

  <section class="report-doc__appendix">…</section>
  <footer  class="report-doc__footer-disclaimer">…</footer>
</article>
Statuswerte für data-report-status.
  • draft — Watermark ENTWURF, nur für interne Vorab-Reviews
  • final — kein Watermark, freigegeben zur Verteilung
  • confidential — Watermark VERTRAULICH, beschränkter Verteiler
  • auditor-only — Watermark AUDITOR-ONLY, nur für externe Auditoren

Report · Executive Summary

Executive Summary

TL;DR-Box mit 3-5 Key-Findings + 4-up-KPI-Grid. Die Box steht vor der ersten Daten-Section: ein Vorstand soll in 30 Sekunden den Status erkennen können. Die TL;DR-Items haben einen tone-Modifier (ok, warn, danger) — der Punkt-Indikator färbt sich entsprechend. 

<section class="report-doc__exec-summary">
  <h2 class="report-doc__exec-heading">Executive Summary</h2>

  <div class="report-doc__tldr">
    <p class="report-doc__tldr-label">TL;DR</p>
    <ul class="report-doc__tldr-list">
      <li class="report-doc__tldr-item report-doc__tldr-item--ok">
        <span class="report-doc__tldr-dot"></span>
        <span><strong>ISMS-Health</strong> — 92% (+4% ggü. Vorquartal)</span>
      </li>
      <li class="report-doc__tldr-item report-doc__tldr-item--warn">…</li>
    </ul>
  </div>

  <div class="report-doc__kpi-grid">
    <div class="report-doc__kpi-tile report-doc__kpi-tile--success">
      <div class="report-doc__kpi-label">ISMS-Health</div>
      <div class="report-doc__kpi-value">92%</div>
      <div class="report-doc__kpi-trend report-doc__kpi-trend--up">▲ +4%</div>
    </div>
    <div class="report-doc__kpi-tile report-doc__kpi-tile--warning">…</div>
    <div class="report-doc__kpi-tile report-doc__kpi-tile--danger">…</div>
    <div class="report-doc__kpi-tile report-doc__kpi-tile--neutral">…</div>

</section>
KPI-Tile-Tones. success (grün) / warning (gelb) / danger (rot) / neutral (primary). Der linke Border-Stripe färbt sich entsprechend. Die --vorstand-Audience hebt die Tile-Höhe und Schriftgrößen an, die --auditor-Audience reduziert sie zugunsten dichterer Tabellen.
Report · Data Sections

Daten-Sections (wiederholbar)

Eine Daten-Section bündelt eine Frage-Antwort-Einheit: Heading + optionaler Intro-Text + Charts (responsive Grid, fallback dompdf-img) + Tabelle (Zebra, Sticky-Header on Web). Mehrere Sections werden hintereinandergehängt — beim Druck setzt CSS einen Page-Break zwischen jeder Section. 

<section class="report-doc__data-section">
  <h2 class="report-doc__section-heading">Risiko-Verteilung nach Score</h2>
  <p  class="report-doc__section-intro">Das Risikoportfolio …</p>

  <div class="report-doc__chart-grid">
    <figure class="report-doc__chart report-doc__chart--bar">
      <img src="…/risk-bar.png" alt="">
      <figcaption class="report-doc__chart-caption">Risiken nach Schweregrad</figcaption>
    </figure>
    <figure class="report-doc__chart report-doc__chart--pie">
      <div class="report-doc__chart-placeholder" data-chart-type="pie">Pie</div>
    </figure>
  </div>

  <div class="report-doc__table-wrap">
    <p class="report-doc__table-caption">Top 10 Risiken nach Inherent-Risk</p>
    <table class="report-doc__table">
      <thead><tr><th>ID</th><th>Titel</th><th>Score</th><th>Owner</th></tr></thead>
      <tbody>…</tbody>
    </table>
  </div>
</section>
Chart-Strategie. Bei Web-Preview rendert die Section statt eines Bildes ein .report-doc__chart-placeholder mit Schraffur — sichtbar für den Designer, ignoriert beim Druck wenn ein image_src übergeben wird. Server-Side rendert z.B. SvgChartRenderer via chart.js-node ein PNG, das dann via dompdf inline eingebettet wird.
Report · Appendix

Appendix — Methodik · Datenquellen · Glossar

Pflicht-Slot für Audit-Festigkeit. Der Auditor will wissen: woher kommen die Zahlen, wie wurden sie berechnet, was bedeuten die Begriffe. Ohne Appendix ist ein Report nicht audit-tauglich — und ein Vorstand traut Zahlen ohne Methodik-Hinweis sowieso nicht. 

<section class="report-doc__appendix">
  <h2 class="report-doc__appendix-heading">Anhang</h2>

  <div class="report-doc__appendix-block">
    <h3 class="report-doc__appendix-subheading">Methodik</h3>
    <div class="report-doc__appendix-body">
      <p>Risikoscore berechnet als Probability × Impact (1-25). Threshold: ≥20 critical, ≥12 high.</p>
    </div>
  </div>

  <div class="report-doc__appendix-block">
    <h3 class="report-doc__appendix-subheading">Datenquellen</h3>
    <ul class="report-doc__appendix-sources">
      <li><strong>ISMS-DB</strong> — Stand 30.06.2026, 23:59 UTC</li>
      <li><strong>Compliance-Engine</strong> — ISO 27001:2022 Annex A</li>
    </ul>
  </div>

  <div class="report-doc__appendix-block">
    <h3 class="report-doc__appendix-subheading">Glossar</h3>
    <dl class="report-doc__appendix-glossary">
      <div><dt>MTTR</dt><dd>Mean Time To Resolve — durchschnittliche Lösungszeit</dd></div>
      <div><dt>ISMS-Health</dt><dd>Composite-Score aus 5 Indikatoren …</dd></div>

  </div>
</section>
Report · Footer / Disclaimer

Footer-Disclaimer

Schließt jeden Report ab. Drei Bereiche: Verteilungsliste (wer bekommt diesen Report?), Vertraulichkeitsklausel (Disclaimer-Box mit Default oder custom-Text), Branding-Footer (Tenant-Name + Generated-At + Klassifikation + Page-Number). Die Page-Number kommt im Print über @bottom-right counter(page) " / " counter(pages).

Report · Audience-Layouts

Audience-spezifische Layout-Varianten

Drei Audience-Modifier passen das Layout an die Zielgruppe an, ohne die Slot-Reihenfolge zu ändern:

--vorstand
Visual-first. Größere KPI-Tiles (110px), Charts in 2-up Grid, weniger Text. „Auf den ersten Blick erkennbar."
--auditor
Table-first. Dichte Tabellen (11pt), kleinere Charts, kompakte KPI-Tiles. „Vollständige Datenbasis."
--aufsicht
Norm-first. Appendix optisch hervorgehoben (3px primary border-top), 2-Spalten-Cover-Grid. „Norm-Anker im Mittelpunkt."
// Wahl per data-Attribut + class-Modifier
<article class="report-doc report-doc--final report-doc--vorstand"
         data-report-audience="vorstand">…</article>

<article class="report-doc report-doc--final report-doc--auditor"
         data-report-audience="auditor">…</article>

<article class="report-doc report-doc--auditor-only report-doc--aufsicht"
         data-report-audience="aufsicht">…</article>
Layout-Auswahl im Controller. Üblicherweise vom Use-Case bestimmt: board_one_pager → vorstand, iso_audit_report → auditor, nis2_quarterly → aufsicht. Wenn der gleiche Bericht für mehrere Audiences gerendert werden muss (z.B. Risk-Report sowohl für Vorstand als auch ISMS-Manager), den Audience-Modifier per Query-Param oder Tab umschalten.
Report · Watermark

Status-Watermarks

Drei Stati zeigen einen Watermark hinter dem Inhalt: draft (rot, „ENTWURF"), confidential (gelb, „VERTRAULICH"), auditor-only (blau, „AUDITOR-ONLY"). final zeigt keinen Watermark — der Bericht ist freigegeben. Implementiert als ::before-Pseudo-Element mit z-index: 1 hinter dem Inhalt (z-index: 2) — Text bleibt selektierbar. 

.report-doc--draft::before { content: attr(data-watermark); color: var(--danger-glow); }
.report-doc--confidential::before { color: var(--warning-glow); }
.report-doc--auditor-only::before { color: var(--primary-glow); }
.report-doc--final::before { /* kein Watermark */ }
Druckverhalten. Der Watermark erscheint im Print proportional zur Größe einer A4-Seite (clamp(48px, 12vw, 140px)). Auf jeder gedruckten Seite — durch das fixed positioning des Pseudo-Elements innerhalb der Article-Box. Wenn der Watermark auf jeder Seite garantiert sichtbar sein muss, einen position: fixed-Trick im @media print Block ergänzen (siehe policy-doc.css für Pattern).
